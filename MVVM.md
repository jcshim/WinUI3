## 💡 MVVM 패턴이란?

**MVVM**은 **Model-View-ViewModel**의 약자로, 소프트웨어 아키텍처 패턴 중 하나입니다. 주로 사용자 인터페이스(UI)를 개발할 때 **UI와 비즈니스 로직을 분리**하여 개발, 테스트, 유지보수를 용이하게 하기 위해 사용됩니다.

특히 **XAML** 기반의 프레임워크(예: WPF, UWP, WinUI 3)에서 데이터 바인딩(Data Binding) 기능을 효율적으로 사용하기 위해 마이크로소프트에서 널리 장려하고 있습니다.

---

### 🧱 세 가지 주요 구성 요소

MVVM 패턴은 이름에서 알 수 있듯이 세 가지 독립적인 구성 요소로 이루어져 있습니다. 

#### 1. Model (모델)
* **역할:** 애플리케이션의 **데이터**와 **핵심 비즈니스 로직**을 담당합니다.
* **포함 요소:** 데이터베이스 접근, 네트워크 통신, 데이터 모델 클래스 (예: `User`, `Product`) 등이 포함됩니다.
* **특징:** UI나 ViewModel에 대해 전혀 알지 못하며, 오직 순수한 데이터와 로직만 다룹니다.

#### 2. View (뷰)
* **역할:** 사용자에게 보여지는 **UI 구성 요소**를 담당합니다.
* **포함 요소:** 버튼, 텍스트 박스, 목록 등 화면에 보이는 모든 XAML 또는 UI 요소.
* **특징:** **ViewModel**에 있는 데이터를 **데이터 바인딩**을 통해 표시하고, 사용자의 입력(클릭, 타이핑)을 **커맨드(Command)**를 통해 ViewModel로 전달합니다. View는 로직을 거의 가지지 않습니다.

#### 3. ViewModel (뷰모델)
* **역할:** **View와 Model 사이의 중개자** 역할을 합니다. View가 필요로 하는 데이터를 준비하고, View의 요청(사용자 액션)을 받아 Model에게 전달합니다.
* **포함 요소:** View에 표시할 데이터 속성(Property), 사용자의 행동에 반응할 커맨드(Command), Model에서 받은 데이터를 View에 맞게 가공하는 로직.
* **특징:** View에 데이터 바인딩을 제공하기 위해 `INotifyPropertyChanged` 인터페이스를 구현하여 데이터가 변경되었음을 View에 알립니다. **UI 컨트롤에 대한 참조를 가지지 않습니다.**

---

### 🤝 데이터 흐름과 장점

#### 데이터 흐름
1.  **View**는 **ViewModel**의 속성에 바인딩되어 데이터를 표시합니다.
2.  사용자가 View에서 버튼을 클릭하면, **Command**를 통해 ViewModel의 메서드를 호출합니다.
3.  **ViewModel**은 필요한 경우 **Model**에게 데이터 처리 또는 로직 실행을 요청합니다.
4.  **Model**은 데이터를 처리하고 결과를 ViewModel로 반환합니다.
5.  **ViewModel**의 데이터가 업데이트되면, `INotifyPropertyChanged`를 통해 **View**에 알리고, View는 자동으로 화면을 갱신합니다.

#### 주요 장점
* **관심사 분리 (Separation of Concerns):** UI(View)와 비즈니스 로직(Model/ViewModel)이 명확하게 분리되어 코드 이해와 관리가 쉬워집니다.
* **테스트 용이성 (Testability):** View가 없이도 ViewModel과 Model을 독립적으로 테스트할 수 있습니다. UI에 종속되지 않은 로직 테스트가 가능해집니다.
* **유연성 (Flexibility):** UI 디자이너와 개발자가 독립적으로 작업할 수 있으며, UI(View)를 변경해도 비즈니스 로직(ViewModel)에는 영향을 주지 않습니다.
